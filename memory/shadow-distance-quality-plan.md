Problem (user-observed, 2026-06-10): large-radius light + occluder. Near the light the shadow is correct (PCSS soft, no acne). The further the occluder, the worse: smoothing disappears, edges become blocky/staircase. Question was architecture vs bug — answer: BOTH, split below. Code: irlite_spotShadow / irlite_pointShadow in Shadres\Modification\Photon\shaders\include\irlite\irlite_lights.glsl (folders refactored 2026-06-11, see [[sync-workflow]]; same inject in the IterationRP copy). See [[shader-shadow-sampling]], bake side [[addon-shadows]].

ARCHITECTURAL (not fixable, only mitigable): perspective shadow maps have fixed ANGULAR resolution -> world texel size grows linearly with distance from the light. texelWorld(point) = 2d/FACE_SIZE = d/256 at MEDIUM(512); texelWorld(spot) = 2d*tan(outer/2)/TILE ~ d/886 at MEDIUM(1024)+60deg (spots ~3.5x denser than points). An occluder 2x further covers 4x fewer texels — silhouette coarsens. Radius itself changes nothing (only the far plane); a big radius just lets the occluder BE far while still brightly lit (Frostbite falloff holds mid-range), making the artifacts visible. Depth precision (32F) and the depth bias formula were checked and are NOT contributors (bias is correctly world-constant).

SAMPLING DEFECTS (fixable — three world-space constants stop working once texelWorld exceeds them):
1. Normal-offset bias = flat 0.05 world (irlite_lights.glsl ~:245 spot, ~:350 point). On MEDIUM point, texel d/256 exceeds 0.05 at d ~= 13 blocks -> the texel-staircase acne the offset exists to kill RETURNS at distance. (Already flagged in [[shader-shadow-sampling]] as the not-done texel-scaling refinement.)
2. PCSS PCF radius collapses sub-texel: penumbraWorld = lightSize*(dRec-dBlk)/dBlk shrinks ~1/d (correct physics — far occluder + near receiver = sharp shadow), and in texture space ~1/d^2. Default IRLITE_SHADOW_SIZE 0.10 + 1-block blocker-receiver gap: the whole 18-tap PCF disk fits inside ONE point-map texel at d ~= 5 blocks -> filter degenerates to a binary hard tap = "smoothing disappears".
3. Blocker search radius shrinks the same way (sub-texel ~25 blocks on point MEDIUM) -> noisy penumbra estimate. Less critical.

FIX PLAN (phases; GLSL edits go to Shadres\Modification\Photon per [[sync-workflow]], mirror to IterationRP inject + patch regen at the end):
- Phase 1 — texel floors (shader-only, biggest win): query map res via textureSize (tileRes = textureSize(irl_spotShadowAtlas,0).x/4; faceRes = textureSize(irl_pointShadowArray,0).x — floors auto-adapt to presets). (a) offset = max(IRLITE_SHADOW_NORMAL_OFFSET, ~1.5*texelWorld(dist)) — kills distant staircase; cost = texel-scale peter-panning at distance, invisible at that texel size. NOTE: spot computes the offset before dist — estimate dist = length(lp-fragWorld) first. (b) penumbra floor ~1-1.5 texels (NDC for spot: texelNdc = 2/tileRes; angular-as-world for point: 2*refDist/faceRes) — PCF never degenerates. (c) same floor on searchNdc/searchRadius. (d) optional: same offset floor in irlite_vlPointStep/irlite_vlSpotStep (quarter-res VL, likely unnecessary). Test matrix: point MEDIUM, big radius, occluder at 4/8/16/32/48 blocks.
- Phase 2 — sub-texel edge smoothness (optional, evaluate after P1): floors keep the filter alive but the edge still quantizes to map texels. Cheap: SpotlightDepthAtlas NEAREST->LINEAR in Java (compare-after-filter, like the point cube already is). Correct: textureGather + manual bilinear weighting of 4 compares per tap (filter-after-compare); atlas variant MUST clamp UV half a texel inside the tile or gather bleeds the neighbor tile.
- Phase 3 — resolution (architectural lever, only if P1+P2 insufficient): immediate = raise preset (HIGH/ULTRA double/quadruple density; VRAM 640MiB/2.5GiB). Real fix = per-light resolution: big-radius lights get bigger tiles / a higher-res cube class (variable atlas allocator + multiple cube arrays + class index in SSBO) — a Java texture/allocator refactor, separate session. Practical guidance now: prefer SPOT for scenario shots needing far shadows (3.5x texel density of point).

Status (all in the PHOTON inject — per user we edit Photon shaders, NOT the IterationRP patch; file Shadres\Modification\Photon\shaders\include\irlite\irlite_lights.glsl, synced to run\shaderpacks\Photon_IRLite):
- Phase 1 DONE — user-verified in game, committed 973d112 (2026-06-10). Spot moved near/far/fY above the offset; tileRes = textureSize(atlas).x/4, texelNdc = 2/tileRes, texelWorld = dist/fY*texelNdc (spot) | 2*dist/faceRes (point, faceRes = textureSize(cubeArray).x); normalOff = max(IRLITE_SHADOW_NORMAL_OFFSET, 1.5*texelWorld) with 0.0 = disabled (no floor); searchNdc/searchRadius floored at 1 texel; penumbraNdc/penumbra floored at 1.5 texels. VL fast steps (irlite_vlSpotStep/vlPointStep) intentionally NOT floored (quarter-res + dither; revisit if beams band near geometry).
- Phase 2 DONE — user-verified in game, committed 9c19e47 (2026-06-10). Scope decision: SPOT ATLAS ONLY — the point cube array is already LINEAR (compare-after-filter per-tap smoothing, acceptable); proper cube gather needs per-face UV/sign-convention weight math for marginal gain, skipped (optional Phase 2b if spot visibly outclasses point). Implementation: new irlite_spotGatherTap(atlasUV, atlasSize, uvMin, uvMax, cmpDepth) = textureGather 2x2 + step(cmpDepth) per texel + bilinear mix of the four BINARY results (filter-AFTER-compare; gather order x=(i0,j1) y=(i1,j1) z=(i1,j0) w=(i0,j0), f = fract(uv*size - 0.5)). Used by the hard path AND all PCF taps. Tile-bleed guard: tap UV clamped half a texel inside the tile (tileMin/tileMax); at the clamp boundary the out-of-tile gather texel gets bilinear weight 0 -> safe. Gather ignores sampler filter state -> atlas stays NEAREST, NO Java change (shader-only, hot-reloadable). Blocker search kept on raw NEAREST texture() (needs exact depths for blocker distance; in-tile by NDC bounds). Quality-0/hard semantics upgraded from aliased binary to 1-texel-antialiased edge.
- Phase 3 not started (resolution levers: preset bump / per-light tile-cube classes).
