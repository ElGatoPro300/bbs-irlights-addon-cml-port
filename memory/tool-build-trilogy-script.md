ГОТЧА ВЫЗОВА — НЕ оборачивать запуск скрипта в 2>&1 (2026-06-21): запуск & build-trilogy.ps1 ... 2>&1 | Select-String ... РЕ-ТРИГГЕРИТ ту же stderr-gotcha НА УРОВНЕ ВЫЗОВА — 2>&1 мержит stderr git checkout внутри скрипта в поток, и под $ErrorActionPreference='Stop' КАЖДАЯ цель «падает» с FAILED: Switched to branch '…'. ГОТЧА ГРЯЗНОГО ДЕРЕВА — addon :TEMP (2026-06-21): pre-flight абортит «Working tree of 'bbs-irlights-addon' is dirty», если в аддоне есть untracked-папка :TEMP (имя с приватным BBS-глифом U+F03A; git status --porcelain -> ?? "\357\200\272TEMP/"). Это побочный продукт gradle-сборки аддона (пустая на первом прогоне -> git её не видит; после сборки в ней появляется файл -> видна). Память [[reference-edit-routing-by-area]]: TEMP/ не коммитить. Фикс НЕ-инвазивный: добавить *TEMP/ в bbs-irlights-addon/.git/info/exclude (локальный, не коммитится, не удаляет папку) -> git status --porcelain снова пуст.

ОБНОВЛЕНИЕ 2026-07-09 (перф + архитектура): скрипт переписан на worktree-изоляцию — каждая цель собирается в одноразовом .build-wt/ (git worktree add --detach на tip ветки), НИКОГДА не переключает ветки в основных чекаутах (собирает с committed tip; незакоммиченные правки основного дерева не трогает), worktree сносится после (gradlew --stop чтобы снять .gradle-локи на Windows, затем Remove-Item + prune). Матрица теперь per-version core: 5 линий (1.20.4/1.21.1/1.21.4/1.21.11/1.20.1), на каждую publishToMavenLocal core (ветка main обслуживает и 1.20.4 и 1.20.1; ветки core 1.21.1/1.21.4/1.21.11), затем потребители build -x test --refresh-dependencies (+ -Pmc для аддона). Итого ~13 gradle-запусков (5 core-publish + 8 consumers), СТРОГО последовательно (version-flat координата irl-core:1.1 в mavenLocal перезаписывается между линиями -> параллелить нельзя). Почему сборка теперь МИНУТЫ, а не ~40-100с из тайминга выше (те цифры от ДО-worktree универсал-core версии, устарели): каждый запуск = свежий worktree (пустой build/ -> ноль инкрементальности) + gradlew --stop на teardown (холодный daemon каждый раз) + --refresh-dependencies (сетевой re-resolve ВСЕХ зависимостей, не только core). Интернет = вторичный фактор при тёплом кэше (~/.gradle/caches ~3.2G прогрет): --refresh-dependencies бьёт по латентности метаданных, не по объёму. Перф-рычаги НЕ включены: org.gradle.caching отсутствует в обоих gradle.properties (build-cache off -> worktree не переиспользуют выходы task), Xmx1G (мал для Loom remap), нет parallel. Ускорение: org.gradle.caching=true (самый большой выигрыш при worktree-схеме), Defender-исключения (BBS root + ~/.gradle + ~/.m2), Xmx 3-4G, -KeepWorktrees для инкрементальных повторов. Диагностика кто-виноват: --profile на одной цели (resolution=сеть vs task execution=CPU/remap) или повтор с -KeepWorktrees. ЗАМЕР 2026-07-09 (линия 1.21.4, Tier0 включён = глобальный ~/.gradle/gradle.properties: caching+parallel+workers.max=16+Xmx8g+ParallelGC): cold 174s -> warm 131s (~25%, delta 43s; core 20->12s, editor 152->117s, editor доминирует). build-cache РАБОТАЕТ (2-й прогон: compileJava/compileClientJava FROM-CACHE), НО remapJar/remapSourcesJar non-cacheable -> executed всегда (основная масса времени); Loom печатает 'Refresh dependencies is in use, loom will be significantly slower' -> --refresh-dependencies тормозит сам Loom, не только сеть; daemon убивается --stop каждый раз (Starting a Gradle Daemon, stopped Daemons could not be reused = холодный старт). Teardown-баг: editor worktree не удаляется полностью ('could not fully remove ... being used by another process' на minecraft-clientOnly yarn jar) -> мусор в .build-wt/, чистить git worktree prune + rm. Вывод: Tier0 ~25% потолок (полный прогон ~30мин -> ~22мин).
