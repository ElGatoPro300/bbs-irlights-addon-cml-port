Фича добавлена 2026-06-18 на irlights branch main. Режим авто-ставит point light на каждый светящийся ванильный блок в радиусе, с hardcoded цветом/радиусом на тип блока, и гарантирует, что светящийся блок не ловит собственный свет в тень. Решения юзера: авто-светы DO cast shadows (ближайшие выигрывают ограниченные слоты); curated table ONLY (без luminance-fallback). Master toggle по умолчанию OFF — экспериментальный opt-in; включение показывает one-time-per-session warning popup («Экспериментальная функция! Производительность не гарантируется»). Два глобальных слайдера масштабируют таблицу: brightness × (autoLightIntensity) и reach × (autoLightReach), применяются в upsert (подхватываются rolling-сканом в ~1s).

Файлы (все redactor если не указано):
- light/auto/BlockLightDefs.java (NEW) — curated IdentityHashMap<Block, Def{rgb,intensity,radius}> для ~50 эмиттеров (torch family, lanterns, glowstone, sea lantern, shroomlight, froglights, end rod, campfires, lava/fire, beacon, conduit, glow lichen, crying obsidian, respawn anchor, candles, lit furnaces, portals). Использовать resolve(BlockState)/paletteCandidate(BlockState) (НЕ get(Block)): светящийся блок горит, только если он в таблице И getLuminance()>0 (так lit/unlit лампы, dead campfires, uncharged anchors, unlit candles self-handle). Redstone dust — НЕ-luminous спец-кейс: ванильный getLuminance() всегда 0 для REDSTONE_WIRE, поэтому резолвится по state.get(RedstoneWireBlock.POWER)>0 -> ОЧЕНЬ слабый красный glow по power (REDSTONE_BY_POWER[1..15], intensity ~0.06..0.18, radius ~2..4.5). Def.shadows=false для него -> PlacedLight.autoShadowEligible -> LightDriver не даёт shadow-слот (слишком тускло для бейка).
- light/auto/AutoLightManager.java (NEW) — INCREMENTAL ROLLING scan (НЕ all-at-once). Каждый client tick двигает chunk-cursor по in-range региону, full-сканит <=CHUNKS_PER_TICK(12) чанков / <=SECTIONS_PER_TICK(16) эмиссивных секций, потом пауза; полный проход (~1s) -> evict не-увиденных -> заново. section.hasAny(luminance>0||redstone) palette pre-filter, затем 16³ inner sweep. WHY incremental (2026-06-18 fix): старый all-at-once скан был O(radius³) и фризил CLIENT THREAD 50–200ms каждый прогон (хуже на scan-radius 96) -> катастрофический EDITOR INPUT lag даже при высоком среднем FPS (фриз != frame-rate). markDirty/throttle больше нет — rolling-проход подхватывает правки за цикл (заодно убран WorldBlockChangeMixin auto-light сигнал = нет redstone-update спама). Keyed by host BlockPos.asLong; РЕЮЗИТ тот же PlacedLight на позицию -> PlacedLight.id стабилен (shadow-кэши key на id -> no thrash). Own list, NOT LightScene -> авто-исключён из LightStore-персиста, editor source list, LightGuideRenderer. nearest(cam,max) = nearest-first capped feed.
- light/LightDriver.java — кормит autos ПОСЛЕ ручных. Резервирует 16 point-shadow cube-слотов (PointShadowArray.MAX_SHADOWS) под ручные POINT-света с тенями, ramps новые auto-shadow first-bakes (AUTO_SHADOW_RAMP_STEP=2/кадр, reset через resetAutoShadowRamp() из GameRendererLightMixin dormant transition), и caps feed под SSBO headroom (MAX_LIGHTS - LightRegistry.getCount()).
- light/shadow/BlockShadowCollector.java + BlockShadowCache.java — self-shadow fix. Скипать ТОЛЬКО host-cell света, когда он эмитит: getOrCompute передаёт hostX/Y/Z = floor(true light pos) в collectForLight(..., hostX,hostY,hostZ), который делает if (x==hostX&&y==hostY&&z==hostZ && state.getLuminance()>0) continue;. Scoped (НЕ global) -> эмиттеры ЕЛЬЗЕВЕР всё ещё кастят тени -> ноль регрессии для ручных. NB: эти два = LOCKSTEP-файлы — идентичная правка зеркалена в BBS-аддон (bbs-irlights-addon/.../light/shadow/{BlockShadowCollector,BlockShadowCache}.java); python irl-core/tools/verify-shadow-lockstep.py = PASS (all 14).
