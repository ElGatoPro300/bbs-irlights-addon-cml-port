IRL-redactor: редактор света не открывается в редакторе реплеев (диагноз, 2026-06-16). Расследование (3 суб-агента, подтверждено исходниками). Относится к [[project-irlite-base-ported]].

Корень. Редактор света = модальный MC Screen (LightEditorScreen), открывался клавишей L ТОЛЬКО когда mc.currentScreen == null. Двойная блокировка:
1. Ванильный MC накапливает KeyBinding.wasPressed() лишь когда не открыт ни один Screen (иначе ввод уходит в открытый Screen) -> пока открыт редактор реплеев, L физически не «нажимается» (IRLRedactorClient.java:55-62, поллинг в ClientTickEvents.END_CLIENT_TICK).
2. Явный гейт if (client.currentScreen == null) client.setScreen(new LightEditorScreen()).
Рендер ImGui (MinecraftClientImGuiMixin -> LightEditorScreen.renderActiveOverlay) рисовал панель ТОЛЬКО если currentScreen instanceof LightEditorScreen. А currentScreen в MC РОВНО ОДИН -> редактор реплеев и наш редактор ВЗАИМОИСКЛЮЧАЮЩИ by design.

Конкретика по двум «реплеям»:
- BBS Films (то, что реально стоит в инстансе C:\prismlauncher\instances\BBS — bbs-2.2-1.20.4.jar): редактор фильма обёрнут в mchorse.bbs_mod.ui.framework.UIScreen extends Screen, т.е. currentScreen != null всё время редактирования -> L мёртв.
- Standalone Replay Mod (johni0702): оверлей playback'а — НЕ Screen в режиме «облёт камерой» (currentScreen==null, курсор захвачен), но превращается в реальный AbstractGuiOverlay$UserInputGuiScreen extends Screen КАК ТОЛЬКО курсор свободен (таймлайн — именно когда юзер хочет открыть панель) -> L мёртв. Плюс Mixin_ContextualKeyBindings переписывает кейбинды в реплее (может «затенить» L). mc.player = CameraEntity extends ClientPlayerEntity (НЕ null). Видео-экспорт рендерит в offscreen VirtualWindow -> пост-blitToScreen ImGui-оверлей+гизмо в итоговое видео НЕ попадут (но свет от пропатченного шейдерпака попадёт — он в world-render, не в оверлее).

Что НЕ ломается: панель null-безопасна (mc.player/mc.world/camera/gameRenderer везде с гардами). Это ЧИСТО «не открывается», НЕ краш.

Прецедент (как было в оригинале): IRLite (аддон BBS) НЕ открывал отдельный Screen — регистрировал свет как BBS-формы и встраивал UI в форм-редактор BBS (UIFormEditorMixin/BBSModMixin), свет кейфреймился как актёр фильма. Поэтому «редактировать свет в редакторе реплеев» РАБОТАЛО — UI света был ЧАСТЬЮ редактора. Standalone-форк намеренно «de-BBS'd» в keybind-Screen -> отсюда конфликт.

План починки (рекоменд. = универсальный): ОТВЯЗАТЬ ImGui-панель от Screen: (а) toggle через сырой GLFW-колбэк/glfwGetKey, независимый от currentScreen -> флаг editorVisible; (б) renderActiveOverlay рисует по editorVisible, а не по instanceof LightEditorScreen; (в) курсор: освобождать при показе, возвращать при скрытии; (г) ГЛАВНАЯ работа — захват ввода: mixin, отменяющий Mouse.onMouseButton/Keyboard.onKey/Screen.mouseClicked когда ImGui.getIO().getWantCaptureMouse()/Keyboard(), чтобы клики не утекали в нижний BBS/Replay Screen.

РЕШЕНИЕ ПОЛЬЗОВАТЕЛЯ (2026-06-16): цель — standalone Replay Mod (johni0702), НЕ BBS Films. Подход = вариант A (отвязать ImGui от Screen): детект реплея через ReplayModReplay.instance.getReplayHandler()!=null (рефлексия, гейтить FabricLoader.isModLoaded("replaymod")); toggle панели через СЫРОЙ GLFW key-callback/glfwGetKey (минует KeyBinding-роутинг и Mixin_ContextualKeyBindings); рендер по флагу editorVisible; ГЛАВНАЯ работа = миксин на Mouse.onMouseButton/Keyboard.onKey, отменяющий ввод MC когда ImGui хочет capture, + примирение курсора в облёте. Вариант B (BBS-форма) к Replay Mod НЕприменим.

В preview MinecraftClient.render+blitToScreen идут штатно -> наш post-blit миксин фаерится. mc.player=CameraEntity extends ClientPlayerEntity. Экспорт = offscreen VirtualWindow. - Фаза 0 (предусловие): определить версию; поднять тестовый инстанс с Replay Mod; Replay Mod НЕ делать build-dep — гейт FabricLoader.isModLoaded("replaymod") + рефлексия.
