Editor/UI on 1.21.11 (in-world-tested fixes, commit 98cb291 — non-obvious):
- ImGui panel was INVISIBLE (screen dims, no UI). Cause: 1.21.11 GUI is DEFERRED — Screen#render only records DrawContext commands; the GuiRenderer flushes them to the framebuffer AFTER render returns, overwriting any raw-GL ImGui drawn inside render. FIX: draw ImGui from a MinecraftClient#render(Z)V mixin injected INVOKE AFTER Framebuffer.blitToScreen() — the whole frame is on the default framebuffer (FBO 0) by then, before swapBuffers, so ImGui lands on top. MinecraftClientImGuiMixin -> LightEditorScreen.renderActiveOverlay() (binds FBO 0 via GL30.glBindFramebuffer(GL_FRAMEBUFFER,0), then ImGuiRuntime.frame(panel::draw)); Screen#render draws nothing now. NOTE: MC's main framebuffer is a GpuTexture (GpuDevice), NOT a bindable GL FBO — you can't draw into it with raw GL; FBO 0 post-blit is the place.
- Line guides crashed: Missing elements in vertex: LineWidth. The 1.21.11 RenderLayers.lines() format is POSITION_COLOR_NORMAL_LINE_WIDTH, so every vertex must call VertexConsumer.lineWidth(w) (in addition to position/color/normal). LightGuideRenderer.line() now does; the whole guide render is wrapped in a latch-off try/catch (a non-critical overlay must never crash the game).
- Menu-background dim+blur behind the editor: in 1.21.11 Screen#renderWithTooltip (final) calls renderBackground SEPARATELY from render (so a no-op render does NOT stop the background — that's why blur/dim persisted). renderBackground (world != null, deferSubtitles() false -> else branch) calls applyBlur(context) then renderDarkening(context) (the latter tiles INWORLD_MENU_BACKGROUND_TEXTURE over the frame). LightEditorScreen now no-ops BOTH applyBlur(DrawContext) AND renderDarkening(DrawContext) -> fully clear, undimmed world view behind the panel (user wanted NO dim on 1.21.11; this drops the 1.20.4 dim look on purpose).
