На этом проекте (IRLite / Photon-порт) естественные точки коммита = CONFIDENCE-чекпоинты: после завершённой и проверенной фазы, либо после проверенного багфикса. В такой точке ПРЕДЛОЖИТЬ коммит (сводка что войдёт) и коммитить ТОЛЬКО после явного подтверждения. Никогда не коммитить/пушить без подтверждения, даже если правки выглядят готовыми.

Why: пользователь хочет, чтобы коммиты отмечали проверенные known-good состояния (каждая фаза [[photon-pipeline]] + каждый багфикс) и сам контролировал, когда пишется история.
How: когда фаза/багфикс достигли «выглядит верно / проверено» — пауза и вопрос про коммит. По подтверждению: репо solo, git user qualet, всё прямо в master (ветвление per harness не нужно — соло-репо). Сообщение в стиле репо (русский, короткий императив, напр. «тени блоков мира: сбор, кэш и инвалидация…») + trailer Co-Authored-By. Один чекпоинт = один сфокусированный коммит, не бандлить несвязанное. В середине фазы — править без коммита.

GITIGNORE-гоча (проверено Phase 1, 2026-06-06, стоило запоротого первого коммита): .gitignore строка ~120 = широкое правило shaders/, поэтому НОВЫЙ файл под Shadres/**/shaders молча пропускается обычным git add (git показывает родит. папку как ?? из-за неигнорируемого соседа типа LICENSE — это МАСКИРУЕТ проблему: git add Shadres/Modification закоммитит только LICENSE). Существующие паки (Shadres/Original/IterationRP/shaders, Shadres/Original/Photon, Shadres/Modification/Photon — папки рефакторнуты 2026-06-11, см. [[sync-workflow]]) ОТСЛЕЖИВАЮТСЯ: предшествуют правилу / force-added; раз файл tracked, правки коммитятся нормально вне зависимости от .gitignore. ПРАВИЛО: правка уже-tracked шейдера = обычный git add; добавление НОВОГО шейдера = git add -f. Всегда git diff --cached --stat перед коммитом пака — убедиться, что застейджены шейдеры, а не только LICENSE. (git add -f на пак даёт один безвредный warning «LF will be replaced by CRLF» на файл — игнор.)
