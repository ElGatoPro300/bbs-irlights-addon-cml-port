## RECOMMENDATION
Рекомендую двухфазный путь с чёткой точкой останова. Фаза 1 = Стратегия A с влитой в неё C (тюнинг-пакет — это естественная часть того же захода в irlite_lights.glsl): baked min/max/avg-пирамида + early-out + адаптивные квантованные тапы + hw-PCF контактной зоны + выравнивание бюджетов к нативным пак-уровням + фиксы перерасходов (BSL-outline fast-path, Photon VL-stride). Это атакует главную метрику (fetches и дивергенция при 10-50 источниках) с сохранением формата карт, контракта SSBO и VL, всех девяти хрупких инвариантов F1-F9 — то есть с минимальным риском дисквалификации по картинке, и при этом эксплуатирует единственное настоящее преимущество проекта над хост-паками: амортизированный бейк. После Фазы 1 замерить (FrameProfiler + профиль GPU-bound сцены): если пенумбра-полоса при высокой Softness всё ещё доминирует в кадре или «зернистый квадрат» (P1) не ушёл — идти в Фазу 2 = Стратегия B (EVSM/MSM-гибрид), начиная со spot-атласа (проще: нет швов куба), где префильтр решает обе проблемы по построению. Важная оговорка по противоречию в досье: audit_issues фиксирует, что PCSS-алгоритм для P1 «эмпирически исключён» (байт-в-байт с работающим эталоном IRLEngine), а audit_glsl называет статичный шум и шумный blocker search сильными кандидатами — эти утверждения конфликтуют (эталон использует ту же математику, но бага не имеет). Пирамидный avg-blocker Фазы 1 и префильтр Фазы 2 закрывают ОБЕ гипотезы, но диагностический шаг «визуальное заземление» из памяти проекта стоит сделать до крупных вложений в Фазу 2.

## RISKS
1) Gutter/padding спот-атласа: mip-уровни пирамиды (и тем более blur-цепочка Фазы 2) протекают между тайлами 4×4 — нужна правка layout'а бейкера и клампов LOD; это единственное место, где Фаза 1 всё-таки трогает Java-контракт. 2) Point cube-array: per-face пирамида ломается на границах граней — обязателен консервативный fallback (полный PCSS у рёбер), иначе швы; для Фазы 2 per-face блюр со швами — известная возня. 3) textureLod для samplerCubeArray в #120-паках (BSL/Bliss) не проверен — рабочая гипотеза «драйверы permissive» может не подтвердиться; закладывать per-pack #define-фолбэк без пирамиды. 4) Пост-бейк пассы (пирамида/блюр) обязаны войти в бюджет allowStaticBake, иначе усугубится открытый спайк C10 ~320ms. 5) Адаптивное ветвление само добавляет дивергенцию — обязательно квантование N в 3-4 ступени и замер до/после, иначе можно съесть выигрыш early-out. 6) Для Фазы 2: light bleeding EVSM в мультислойных интерьерах Minecraft (типовой кейс проекта) + память ×2-4 поверх уже недокументированного юзеру удвоения static-слоем (ULTRA ~5GiB) — HIGH/ULTRA пресеты потребуют пересмотра форматов (fp16/MSM-16bit), и это единственный пункт с реальным риском «заметной деградации = дисквалификация». 7) Контрактное противоречие досье: research_pcss рекомендует temporal-джиттер под TAA, но канон проекта (shader-shadow-sampling) явно фиксирует spatial-only ротацию как инвариант («анти-паттерн = frameCounter-grain») — temporal-опцию вводить только отдельным opt-in #define после явного пересмотра этого инварианта, не по умолчанию.

## RANKING
### [9] Стратегия A — «Пирамидный PCSS»: baked min/max/avg mip-пирамида карт + early-out (1 fetch вне пенумбры) + адаптивные тапы (3-4 квантованные ступени, hw-PCF при penumbra<1.5 texel) + Vogel+IGN highp + bias-пакет и rUV-клампы к тайлу
Лучшее соотношение выигрыш/риск по перф-линзе. Использует уникальный рычаг проекта (бейк амортизирован): пирамида строится один раз на dirty-бейке, в кадре 1-2 textureLod закрывают 70-90% (light,pixel)-пар до входа в PCSS — прямой удар по главной метрике (fetches при 10-50 источниках). avg-канал заменяет 6-20-таповый blocker search одним fetch'ем и заодно стабилизирует шумную оценку penumbra (кандидат №1 на «зернистый квадрат» по audit_glsl п.4). Квантованные ступени N ограничивают дивергенцию варпов (сейчас — основной GPU-bound фактор, audit_glsl п.5). КРИТИЧНО: не меняет формат depth и контракт SSBO/VL вообще — волюметрика Photon даже выигрывает (1D min/max skip, Chen 2011); нулевой риск деградации картинки (чистый акселератор); хрупкие инварианты F1/F8/F9 не трогаются. Минус: потолок качества остаётся PCSS'ным — широкая Softness всё ещё N тапов в полосе пенумбры; зерно уменьшается, но не исчезает по построению.

### [8] Стратегия B — «Префильтр-гибрид»: EVSM (spot: EVSM4/RGBA16-32F, point: MSM RGBA16-UNORM или EVSM4 fp16) + Gaussian-blur+mip-цепочка на бейке + VSSM-оценка блокера 1 fetch'ем + контактная PCF-ветка по сырой depth; конверсионный пасс depth->moments после бейка (кастер-шейдеры не трогаются)
Максимальный потолок по перф-линзе: 2-4 fetch'а на (свет,пиксель) при ЛЮБОЙ Softness, стоимость перестаёт расти со слайдером; детерминированный трилинейный результат без стохастики = нет дивергенции по путям шума и нет «зернистого квадрата» по построению (закрывает P1 независимо от того, бейк это или сэмплинг — блюр бейка замажет и бейк-сторону); mips = анти-алиасинг на дали (P7); Чебышёв без depth-compare ослабляет bobbing-acne (P2); VL — 1 мягкий fetch на шаг без bias-борьбы. Бейк-цена амортизирована кэшем (уникально для этого проекта — у шейдерпаков так нельзя). Почему не первое место: объём и риск заметно выше A — память ×2-4 на уже тяжёлых пресетах (HIGH 640MiB -> потенциально >1.5GiB с учётом static-слоя ×2, аудит Java §5), light bleeding в мультислойных Minecraft-интерьерах (мебель/пол/потолок — худший кейс VSM-семейства), gutter/mip-clamp атласа = правка layout-контракта, per-face блюр куба со швами, непроверенный textureLod для samplerCubeArray в #120-паках, и смена семантики карты = миграция всех 7 инжектов + VL-контракта разом.

### [7] Стратегия C — «дешёвый тюнинг-пакет» (без смены архитектуры): early-out по первым 4 тапам (Photon-паттерн), fast=true для тусклых/дальних источников (attenuation-LOD), fast=true в BSL outline-пассе, hardware sampler2DShadow/samplerCubeArrayShadow как второй сэмплер для PCF-фазы (blocker остаётся raw), выравнивание бюджетов к нативным (blocker 4-6/PCF 9-16), point gather/сглаживание, backport VL-stride в Photon
Самый быстрый ROI: всё — include-уровень, ни одного изменения Java/форматов/контрактов. Досье даёт готовые цифры: хост-паки живут на 3-6 blocker + 9-16 PCF тапах против наших 10+18 (default) — значит default-пресет можно резать почти вдвое без потери паритета с «художественным уровнем» паков; early-out экономит «в разы» на 90% экрана (Photon-прецедент); удвоение PCSS в BSL-outline и VL без stride в Photon — чистые баги-перерасходы (audit_glsl п.7, п.10). hw-PCF = бесплатный ×4 на тап, срезает тапы ещё вдвое на LOW/MED и чинит несогласованность point vs spot (audit_glsl п.2). Но потолок низкий: цикл по светам, дисперсия при высокой Softness и «квадрат» остаются; это не рефакторинг, а гигиена — правильна как фаза 0 внутри A. Оговорка досье: два сэмплера на одну depth-текстуру (raw + compare) требуют texture views/sampler objects — небольшой Java-кусок всё же есть.

### [4] Стратегия D — screen-space shadow blur как LOW/MEDIUM-пресет: 1 hard-tap на свет -> агрегированный IRLite-colortex + penumbra-ширина -> separable cross-bilateral blur пасс
Асимптотически лучший ответ на 10-50 источников (blur = фиксированный fullscreen-пасс, per-light падает до 1-2 тапов), и «квадрат» исчезает по построению. Но по совокупности дисквалифицирующих факторов проигрывает: (1) pass-уровень = хирургия +colortex +composite-проход в КАЖДОМ из 7 паков (масштаб VL-порта ×7, против include-уровня у A/B); (2) агрегированный буфер = кросс-блид пенумбр разноцветных перекрывающихся источников — а это ровно целевые сцены проекта (десятки цветных ламп), т.е. реальный риск «заметной деградации = дисквалификация»; (3) VL продолжает сэмплить сырые карты — мягкость поверхности и god-rays визуально расходятся; (4) low-res penumbra-mask даёт swimming bubbles (отмечено в самом research). Допустима позже как опт-ин пресет, не как магистраль.

### [3] Стратегия E — гибрид SDF: baked каскадный 3D-SDF мира (блоки) со sphere-trace пенумброй + существующие карты только для динамики (актёры/кости/реплеи)
Стратегически красиво (память не растёт с числом источников, нет acne по построению, качество не падает с дистанцией, VL упрощается), но по текущей перф-линзе выигрыша не гарантирует: 8-24 шага sampler3D на КАЖДЫЙ свет на пиксель — сравнимо с пост-A PCSS, а sphere-trace по своей природе дивергентен (длина марша зависит от геометрии на пиксель) — бьёт ровно в главную метрику «дивергенция при 10-50 источниках». Плюс гибрид означает ДВЕ системы на пиксель (SDF-трейс + map-сэмплинг для динамики) — суммарная стоимость может вырасти. Качество: SDF-разрешение <1 блока размывает заборы/листву/BBS-декор — риск заметной деградации именно в кино-сценах проекта. Объём работ максимальный (генерация, каскады, инвалидация, новый контракт). Это отдельный будущий проект, не рефакторинг фильтрации.

### [2] Стратегия F — PCSS + screen-space contact shadows (IRLITE_CONTACT_SHADOWS, только deferred-паки Photon/IterationRP)
По перф-линзе — чистый минус: ДОБАВЛЯЕТ 8-16 depth-тапов на источник на пиксель поверх существующего PCSS, ничего не убирая; сам research признаёт «НЕ для десятков источников» (нужен отбор 1-4 доминантных). Покрытие — только 2 пака из 7 (в forward-паках depth не полон в light-loop), т.е. ломается принцип единого поведения инжекта. Решает нишевую проблему (peter-panning контактной зоны от normal-offset), которая в каталоге проблем проекта даже не значится среди открытых. Как когда-нибудь-опция для Photon — может быть; как стратегия рефакторинга — нет.
