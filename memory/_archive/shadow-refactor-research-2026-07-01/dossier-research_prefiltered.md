# Пре-фильтруемые карты теней (VSM / ESM / EVSM / MSM) для IRLite/irl-core

## 0. TL;DR под наш контекст

Все четыре техники устроены одинаково по скелету: вместо depth-теста «z < d?» в карту пишется **фильтруемая статистика глубины** (моменты или экспонента), которую можно блюрить, мипмапить и сэмплить аппаратной би/трилинейкой — а shadow-тест восстанавливается из отфильтрованных значений аналитически. Классический трейд-офф «дороже на стороне карты (blur+mips каждый кадр), дешевле на стороне пикселя» у нас ломается в лучшую сторону: **карты бейкаются один раз и кэшируются, значит blur+mip-цепочка амортизируются почти в ноль**, а per-pixel стоимость падает с 32–80 фетчей PCSS (blocker search + variable PCF) до **1 трилинейного фетча + немного ALU**. Для GPU-bound сцен с десятками источников это лучший из известных рычагов. Плата: память ×2–4 на тексель, light bleeding (лечится, но не до нуля), и переделка контракта сэмплинга (включая волюметрику — которая, впрочем, выигрывает сильнее всех).

---

## 1. VSM — Variance Shadow Maps (Donnelly/Lauritzen; GPU Gems 3 ch.8)

**Математика.** Хранятся два момента распределения глубины по фильтр-окну: M1 = E[z], M2 = E[z²]. После любой линейной фильтрации восстанавливаем среднее μ = M1 и дисперсию σ² = M2 − M1². Тест — верхняя граница по одностороннему неравенству Чебышёва:

```
p_max(d) = σ² / (σ² + (d − μ)²)   при d > μ, иначе 1.0
```

`p_max` — доля незатенённости. Так как это линейная статистика, **любой линейный фильтр (Gaussian blur, mip, bilinear, anisotropic, MSAA) применим к карте напрямую** — уникальное свойство всего семейства.

**Формат.** RG32F — эталон (Лауритцен прямо рекомендует fp32 + **линейную метрику глубины** вместо post-projection z). RG16F страдает квантованием (шахматный acne); RG16 **UNORM с линейной глубиной** работает лучше, чем fp16 (наблюдение MJP). Итого 8 байт/тексель (fp32) против 2–3 у raw depth.

**Стоимость сэмпла.** 1 билинейный (2 при трилинейке — аппаратно всё равно «1 fetch» для шейдера) + ~5–8 ALU (Чебышёв). Против текущего PCSS это уменьшение на 1.5–2 порядка числа обращений к текстуре.

**Артефакты.**
- **Light bleeding** — главный: при нескольких разрывах глубины в окне фильтра (occluder A → occluder B → receiver) Чебышёв даёт завышенный `p_max`, светлые «протечки» в глубокой тени. Растёт с ростом отношения (расстояние A–B)/(B–receiver).
- Numerical: при σ²→0 нужен клампинг минимальной дисперсии (min-variance bias), иначе шум.

**Фиксы.** *Light bleeding reduction* Лауритцена: `p_max' = linstep(bleedReduction, 1.0, p_max)` — срезает хвост малых значений и перенормирует; дёшево, но переусиливает/утемняет пенумбру. Радикальные фиксы — EVSM/MSM (ниже) и Layered VSM (разбиение диапазона глубины на слои — для нас неинтересно, много памяти/каналов).

## 2. ESM — Exponential Shadow Maps (Annen et al. GI'08; параллельно Salvi)

**Математика.** В карту пишется exp(c·z) (или c·z с фильтрацией в log-пространстве), тест:

```
visibility = saturate( exp(-c·d) · E[exp(c·z)] )  =  saturate( exp(c·(z_filtered − d)) )
```

Работает как «мягкая ступенька»: чем больше c, тем ближе к жёсткому тесту. Типичный c ≈ 80 для fp32.

**Формат.** **R32F, один канал — самая дешёвая по памяти техника семейства** (4 байта/тексель). R16F непригоден (максимум fp16 = 65504 → c ≤ ~11, слишком мягкая ступенька, массивные протечки).

**Стоимость сэмпла.** 1 фетч + exp + saturate (~3 ALU). Абсолютный минимум.

**Артефакты.** Аппроксимация валидна при допущении «receiver локально планарен и глубже блокеров»; при больших ядрах фильтра допущение ломается: незатенённые области у контактов **пересветляются** (протечка у силуэта), Annen для этих случаев предлагал классификацию + fallback на PCF (что убивает нашу цель) либо жить с артефактом. Переполнение exp(c·z) в fp32 при c·z > ~88 — лечится линейной z ∈ [0,1] и c ≤ 80–88, либо log-space фильтрацией.

**Наш вывод по ESM:** как основная техника для прямого света — слабее EVSM/MSM, но это **лучший кандидат для волюметрики** (1 канал, 1 фетч, самый дешёвый инкремент на шаг ray-march; именно ESM исторически популярен для volumetric/hair shadows).

## 3. EVSM — Exponential Variance Shadow Maps (Lauritzen & McCool, Layered VSM paper)

**Математика.** VSM поверх экспоненциально «разогнутой» глубины. Два варианта:
- **EVSM2**: моменты от exp(+c·z) → 2 канала. Убирает bleeding там, где разгиб растягивает разрыв, но протекает у дальних receiver'ов.
- **EVSM4**: моменты от exp(+c₊·z) И от −exp(−c₋·z) (негативный warp), берётся min двух Чебышёвых → 4 канала. Практический стандарт качества (использовался в целом ряде движков как «дефолтная» фильтруемая тень).

**Формат и лимиты экспонент.** Ограничение — квадрат момента должен влезать в формат:
- **fp32**: c₊ ≤ ~42 (exp(42)² ≈ 2.9e36 < 3.4e38). Рабочие значения: c₊ = 40–42, c₋ = 5–10. EVSM4 = **RGBA32F, 16 байт/тексель**.
- **fp16**: c ≤ ln(65504)/2 ≈ **5.54** — warp слабый, bleeding заметно хуже, и bias приходится задирать (MJP: bias 0.0001 достаточно на fp32, до 0.005 нужно на fp16, и рост bias сам возвращает bleeding). RGBA16F = 8 байт/тексель — компромисс «памяти как RG32F-VSM, качество между VSM и EVSM4-fp32».

**Стоимость сэмпла.** 1 фетч + 2×Чебышёв + 2×exp (~15–20 ALU). Всё ещё копейки против фетчей PCSS.

**Артефакты/фиксы.** Наследует минимальную дисперсию и linstep-редукцию от VSM (обычно с куда меньшими порогами); depth-dependent bias в warped-пространстве (производная exp растёт с z — bias надо масштабировать на c·exp(c·z), это 2 строки). По сумме свойств «качество/простота кода» — **сладкая точка семейства**: поверх готового VSM-кода это +15 строк GLSL.

## 4. MSM — Moment Shadow Maps (Peters & Klein, I3D 2015)

**Математика.** Хранятся 4 степенных момента z, z², z³, z⁴; на сэмпле решается задача моментов (через разложение Холецкого 3×3 + корни квадратного/кубического уравнения) и вычисляется **точнейшая возможная нижняя граница** видимости для данных 4 моментов (Hamburger bound; есть чуть более дорогой Hausdorff-вариант, дающий немного другой компромисс протечек). Заявка авторов: «arguably the best possible shadow mapping technique using 64 bits per texel».

**Формат.** Killer-фича: **оптимизированное аффинное преобразование моментов позволяет хранить их в RGBA16 UNORM = 64 бита/тексель** — та же память, что RG32F-VSM, при радикально меньших протечках. fp32-вариант (RGBA32F) ещё чище, но обычно не нужен. Moment bias: ~3e-5 для 16-бит, ~1e-6–3e-6 для fp32 (числа из демо MJP). 16-битный вариант оставляет лёгкий квантовый шум и слабое протекание в short-range тенях — лечится в «Improved MSM» (Peters & Münstermann, JCGT 2017) и нелинейным квантованием (HPG 2017).

**Стоимость сэмпла.** По-прежнему **1 фетч**, но ALU заметно жирнее: ~50–70 инструкций (декодирование квантования + Холецкий + корни). На современном GPU при GPU-bound по текстурным фетчам это почти всегда всё равно выигрыш, но в волюметрике (фетч на каждый шаг марша) MSM-математика на шаг уже ощутима.

**Артефакты.** Минимальный bleeding в семействе при 64bpp; квантовый шум 16-бит версии; требует аккуратной численности (клампы дискриминантов). Кода — ~40–60 строк GLSL, все операции доступны и в старых версиях языка (только арифметика).

## 5. Variable penumbra / contact hardening на пре-фильтре

Пре-фильтр по построению даёт **фиксированный** радиус мягкости (радиус бейк-блюра). Переменная пенумбра получается двумя путями:

### 5.1. Mip-цепочка + выбор LOD по blocker distance (рекомендуемый для нас)
1. На бейке: базовый Gaussian blur (минимальная мягкость) → полная mip-цепочка (каждый mip — ещё более широкий фильтр). Лучше строить mips отдельными down-sample+blur проходами, а не glGenerateMipmap box-фильтром: чистый box-mip даёт «boxy»-артефакты при переходах LOD (известная жалоба по mip-blur VSM), Gaussian-цепочка + трилинейка между уровнями — гладко.
2. В кадре: оценка средней глубины блокера **без blocker search**. Трюк VSSM (Yang et al., PG 2010, построен ровно на этой идее): из тех же моментов
   `z_avgBlocker ≈ (E[z] − p_max·d) / (1 − p_max)` — E[z] и p_max берём одним фетчем грубого mip-уровня (широкое окно). Это замена 16-сэмплового blocker search **одним фетчем**.
3. Пенумбра по обычной PCSS-формуле `w = (d − z_b)/z_b · w_light`, → `lod = log2(w / texel)` → `textureLod(...)`. Итого contact hardening за **2 фетча на пиксель** (1 на blocker-оценку + 1 трилинейный на тень).

### 5.2. SAT-VSM (GPU Gems 3 ch.8, Lauritzen)
Summed-area table по моментам: точный box-фильтр **произвольного** прямоугольного размера за 4 фетча (по углам). Плюс: нет квантования по LOD, идеальная плавность hardening'а. Минусы: SAT съедает точность (~log2(w·h) бит, на 512² это ~18 бит — нужен int32-SAT c wraparound или распределение точности по каналам), память fp32 обязательна, и **на cube-фейсы SAT ложится плохо** (пофейсные SAT + швы). У нас SAT-генерация была бы бесплатной (бейк), но из-за point cube-array и сложности единого GLSL — рекомендую mip-путь, SAT держать как опцию ULTRA только для spot-атласа. VSSM показывал на порядок быстрее PCSS при больших пенумбрах.

Практическое следствие для известного бага: **«зернистый квадрат» при высокой Softness исчезает по построению** — в пре-фильтре нет стохастического blocker search / poisson-PCF, результат детерминированный и гладкий (трилинейка). Если источник зерна — бейк-сторона, Gaussian-блюр бейка тоже его замажет.

## 6. Point-источники: поведение на cube-map array

- **Метрика глубины**: линейное расстояние до света (обычно уже так) — обязательна для VSM/EVSM (Лауритцен: linear depth metric, не post-projection z).
- **Блюр на бейке**: блюрить по-фейсно нельзя наивно — вылезут швы на рёбрах. Корректный способ: для каждого текселя фейса реконструировать направление на сфере и брать соседей сэмплами **самого кубмапа** по сфере (см. kosmonaut blog / gamedev-обсуждения). Так как это бейк — стоимость не важна, а качество швов полное.
- **Сэмплинг в кадре**: включить `GL_TEXTURE_CUBE_MAP_SEAMLESS` (core GL 3.2) — иначе билинейка на рёбрах ломается (классический источник швов, см. кейс Urho3D). Cube-array + mips + textureLod поддерживаются (`GL_ARB_texture_cube_map_array`, GL 4.0 core — у нас уже используется во всех паках).
- Форматы: cube array RGBA32F (EVSM4) тяжёл (6 фейсов × слои × 16 Б); для point-линии разумнее **RGBA16F EVSM4 (c≈5.5)** или **RGBA16-UNORM MSM (64bpp)** — MSM здесь особенно к месту, т.к. при 64bpp даёт лучшее качество.
- Дуал-параболоид как альтернатива кубу (меньше памяти/швов) — не советую: ломает существующий контракт PointShadowArray и добавляет искажения.

## 7. Spot-атлас: специфика

Blur + mips в 2D-атласе требуют **gutter/padding между тайлами** (иначе блюр и трилинейка тянут глубину соседнего источника) и **clamp LOD** на уровень, где тайл ещё ≥ нескольких текселей; UV-клампинг внутри тайла с полутексельным отступом на каждом mip. Это главный интеграционный кусок работы в SpotlightDepthAtlas (не математика, а раскладка).

## 8. Требования GL/GLSL — совместимость с #version 120 паками

- Форматы RG32F/RGBA32F/RGBA16F/RGBA16: `ARB_texture_rg` + `ARB_texture_float` (GL 3.0); **линейная фильтрация fp32** — на всех DX10+ GPU (наша база GL4 — ок).
- Явный LOD во фрагменте на #version 120: `#extension GL_ARB_shader_texture_lod : enable` → `texture2DLod/textureCubeLod` — определены даже внутри неоднородного control flow; расширение вездесущее. Bias-вариант `texture2D(s, uv, bias)` есть в самом core-120, но для penumbra-LOD нужен именно абсолютный LOD, т.е. *Lod-функции.
- `textureLod` для **samplerCubeArray** формально приходит с `GL_ARB_texture_cube_map_array` (написан против GLSL 4.00), но раз cube-array сэмплинг в 120-паках уже работает через расширения (драйверы permissive) — рабочая гипотеза: и Lod-оверлоады доступны; это надо проверить на месте и при необходимости спрятать в per-pack #define-дельту (как сейчас).
- Генерация mips на бейке: `glGenerateMipmap` (GL 3.0) либо свои down-sample проходы (лучше, см. §5.1). Работает и для cube array.
- Вся математика (Чебышёв, exp, MSM-Холецкий) — чистые ALU, никаких требований к версии GLSL. Единый файл `irlite_lights.glsl` с #define-дельтами сохраняется; дельты только вокруг имён Lod-функций (120 vs 330+).

## 9. Ключевой угол: что меняет «бейк один раз + кэш»

| Классический минус техники | Обычно | У нас |
|---|---|---|
| Blur+mip каждый кадр на каждый свет | Главная статья расхода (MJP: EVSM4-fp32 с полным пре-фильтром +11.5ms против 7×7 PCF в его сцене) | **Амортизирован в ноль** (одноразово на бейке; на фоне спайка бейка ~320ms separable blur+mips на 1024² — единицы ms) |
| Память ×2–4 на тексель | Болезненно при каскадах на весь экран | Умеренно: атлас 2048² RG32F+mips ≈ 43 МБ; смягчается MSM-16bit/RGBA16F |
| Light bleeding | Остаётся | Остаётся (единственный настоящий минус); лечится EVSM4/MSM + linstep |
| Стоимость сэмпла | 1 фетч — и так был плюс | **Главный выигрыш**: 1–2 фетча vs 32–80 у PCSS при GPU-bound |
| Волюметрика | По фетчу на шаг — уже хорошо | Ещё лучше: можно брать высокий mip (шире фильтр = глаже VL, меньше кэш-промахов); ESM-канал на шаг = 1 фетч + 3 ALU |
| Bias/acne | Малый bias — плюс | Потенциально закрывает **Photon view-bobbing acne**: тест не бинарный, normal-offset почти не нужен |

Т.е. **бейк-кэш инвертирует главный аргумент «против»** пре-фильтруемых карт и оставляет им только память и bleeding. Дополнительно закрываются или ослабляются три известные проблемы проекта: зернистый квадрат Softness (детерминированный фильтр), качество на дали (trilinear mips = встроенный shadow anti-aliasing на расстоянии), GPU-bound при многих источниках (радикальное падение фетчей).

## 10. Рекомендация для IRLite

1. **Spot-атлас**: EVSM4 RGBA32F (c₊=42, c₋=8) + Gaussian-mip-цепочка; пресеты LOW–ULTRA = формат (RGBA16F/RGBA32F) × наличие contact hardening; ULTRA-опция — SAT-VSM.
2. **Point cube-array**: MSM RGBA16-UNORM (64bpp, лучший на эту память) либо EVSM4 RGBA16F, seamless-cubemap, сферически-корректный бейк-блюр.
3. **Contact hardening**: mip-LOD по VSSM-оценке blocker distance из E[z] (2 фетча/пиксель).
4. **Волюметрика**: тот же ресурс, фетч mip-уровня повыше; при желании — отдельный ESM-канал (R32F) как ультрадешёвый VL-путь.
5. Первый риск-пункт для прототипа: gutter/mip-clamp атласа и проверка cube-array `textureLod` в #version 120 паках.
